51  /**
52   * The main interface to Trap, a TrapEndpoint is the shared interface between servers and clients. It provides most of
53   * the methods that should be necessary to configure and use a Trap connection.
54   * <p>
55   * In general, a TrapEndpoint should be configured either when it has just been created, using a {@link TrapFactory}, or
56   * when it has just been provided to a {@link TrapListenerDelegate} using the callback. Reconfiguring a TrapEndpoint
57   * while it is in use can have unintended consequences.
58   * <p>
59   * Common tasks on a TrapEndpoint are:
60   * <h3>Configuring Transports</h3>
61   * <p>
62   * Enabling and disabling transports, as well as configuring each transport, is possible from TrapEndpoint. Adding new
63   * transports to an existing endpoint is not possible; for that, instead use
64   * {@link TrapTransports#addTransportClass(Class)} before instantiating an endpoint.
65   * <h3>Sending and Receiving Messages</h3>
66   * <p>
67   * To send messages, simply use {@link #send(byte[])} to send binary data, or {@link #send(TrapObject)} to use deferred
68   * serialization. When data is received, the appropriate delegate that implements {@link OnData} or {@link OnObject} is
69   * called.
70   * <h3>Checking Liveness</h3>
71   * <p>
72   * Trap provides a simple facility to check if the other endpoint is alive, that is, communication is active and the
73   * other application layer is responding. The {@link #isAlive(long)} method will perform an active check, that is, ping
74   * immediately. The {@link #isAlive(long, boolean, boolean, long)} method on the other hand can be used to check if the
75   * endpoint has been alive recently, or a combination of the two.
76   * <h3>Configuring Keepalives</h3>
77   * <p>
78   * By default, Trap will attempt to use a per-transport keepalive policy that strikes a moderate balance between
79   * liveness and chattiness. It will take into account traffic only on the current TrapTransport, and without using
80   * device integration. This simple implementation can be tweaked to use a static keepalive interval instead (every X
81   * milliseconds), or disabled, using the {@link #setKeepaliveInterval(int)} method. More advanced keepalives can be
82   * user-supplied on a transport basis using {@link TrapTransport#setKeepalivePredictor(TrapKeepalivePredictor)}.
83   * <h3>Customizing the Message Queue</h3>
84   * <p>
85   * Trap has a number of buffers it uses, and some (such as the message queue) can impact performance significantly under
86   * different usage patterns. The message queue is the first front-end that feeds Trap, and plays a large role. An
87   * endpoint can be instructed to either choose the "best" buffer available of a given type (see constants) using
88   * {@link #setQueueType(String)}, or it can be told to explicitly use a specific queue using
89   * {@link #setQueue(MessageQueue)}. User-supplied queues can be use, as long as they fulfil the requirements.
90   * 
91   * @author Vladimir Katardjiev
92   * @since 1.0
93   */

265     /**
266      * Attempts to verify that the endpoint is alive, with a round trip time of at most <i>timeout</i> milliseconds.
267      * This function is primarily intended for when an application requires to know it has connectivity <i>right this
268      * instant</i>, and should be used sparingly on power-constrained devices.
269      * <p>
270      * {@link #isAlive(long)} is synonymous to calling <i>isAlive(timeout, true, true, timeout)</i>.
271      * <p>
272      * Note that isAlive is intended to be used in an asynchronous mode. The callback may be used to request a
273      * synchronous answer, but this is strongly discouraged. Further note that isAlive will take longer time to execute
274      * than <i>timeout</i> as it will both try to verify transport liveness, and then reconnect the transport if
275      * verification fails. Transport reconnection uses a timer longer than <i>timeout</i>, to allow for additional
276      * round-trips used when connecting.
277      * <p>
278      * <b>Warning:</b> Calling <i>isAlive</i> on a Server Trap Endpoint (i.e. when none of the transports can perform
279      * the open() function) may cause an endpoint to close on the server, while the client considers it open. The client
280      * may not have discovered the underlying transport is dead yet, and may require more time to reconnect. A wakeup
281      * mechanism can be used to establish liveness, but the server's <i>timeout</i> value should be significantly more
282      * generous in order to accommodate the client's reconnect and/or wakeup procedure!
283      * 
284      * @param timeout
285      *            The timeout (in milliseconds) for the round-trip between the server.
286      * @return A callback for <i>true</i> if the connection is currently alive (including if this function successfully
287      *         re-established the connection), <i>false</i> otherwise.
288      * @since 1.2
289      */
290     public Callback<Boolean> isAlive(long timeout);

292     /**
293      * Attempts to verify if the endpoint is alive, or has been alive within a certain number of milliseconds.
294      * Effectively, this can be used to trigger a keepalive check of the endpoint if used with a <i>within</i> parameter
295      * of 0 and a <i>check</i> parameter of true.
296      * <p>
297      * This function has a two-part purpose. The first is for the application to be able to check the last known
298      * liveness of the endpoint, to reduce the discovery time of a dead connection. The second is to trigger a check for
299      * a dead endpoint, when the application needs to know that it has active connectivity.
300      * <p>
301      * Note that in normal operation, the endpoint itself will report when it has disconnected; the application does not
302      * need to concern itself with this detail unless it specifically needs to know that it has connectivity right now.
303      * <p>
304      * <b>Warning:</b> Calling <i>isAlive</i> on a Server Trap Endpoint (i.e. when none of the transports can perform
305      * the open() function) may cause a client to lose its connectivity. The client may not have discovered the
306      * underlying transport is dead yet, and may require more time to reconnect. A wakeup mechanism can be used to
307      * establish liveness, but the server's <i>timeout</i> value should be significantly more generous in order to
308      * accommodate the client's reconnect and/or wakeup procedure!
309      * 
310      * @param within
311      *            Within how many milliseconds the last activity of the endpoint should have occurred before the
312      *            endpoint should question whether it is alive.
313      * @param check
314      *            Whether the endpoint should attempt to check for liveness, or simply return false if the last known
315      *            activity of the endpoint is not later than within.
316      * @param force
317      *            Whether to attempt to force reconnection if the transports are not available within the given timeout.
318      *            This will ensure available liveness value reflects what is possible right now, although it may mean
319      *            disconnecting transports that still may recover.
320      * @param timeout
321      *            If check is true, how many milliseconds at most the liveness check should take before returning false
322      *            anyway. The application can use this value if it has a time constraint on it.
323      * @return A callback for <i>true</i> if the connection is currently alive (including if this function successfully
324      *         re-established the connection), <i>false</i> otherwise.
325      * @since 1.2
326      */
327     public Callback<Boolean> isAlive(long within, boolean check, boolean force, long timeout);

376      * @throws TrapException
377      *             if the queue length is exceeded, or a timeout occurs on a blocking queue
378      */

391     /**
392      * Attempts to queue an object for sending. If the queue length is exceeded, it may block or throw an exception, as
393      * per the queue type.
394      * <p>
395      * Sending a TrapObject will serialize it as late as possible, or never. If the object is sent locally, it will
396      * never be serialized. If it is to be sent remotely, it will be serialized when needed. Trap may serialize multiple
397      * TrapObjects in parallel, so implementations should not assume any order between eventual
398      * {@link TrapObject#getSerializedData()} calls.
399      * <p>
400      * Note that, since the TrapObject is not necessarily serialized in the caller's thread, exceptions caused by object
401      * serialization will most probably not be seen in this way. They will be logged, however.
402      * 
403      * @param object
404      *            The object to send.
405      * @param channel
406      *            The channel to send the data on. Recommended in the range [1-63], but will accept ID zero as well.
407      * @param useCompression
408      *            <i>True</i> if Trap should apply compression algorithm to this message, <i>false</i> otherwise.
409      *            Compression will be applied in the event that the object has to be serialized.
410      * @throws TrapException
411      *             If the object could not be queued for sending, either due to user configuration or because of an
412      *             invalid state of the endpoint.
413      */
414     public abstract void send(TrapObject object, int channel, boolean useCompression) throws TrapException;

416     /**
417      * In async mode, the endpoint will automatically schedule incoming messages on a separate thread than the one on
418      * which they are received. Note that this does not imply there will be multiple threads receiving messages from the
419      * TrapEndpoint, only that there is a break in the thread.
420      * <p>
421      * Async mode is generally preferred over synchronous mode, as it decouples the network layer from the request
422      * processing layer. If the application using Trap provides its own thread dispatching with little overhead
423      * (computational time) after receiving a message from Trap, it may experience a performance gain from switching off
424      * async mode.
425      * <p>
426      * <b>Note that Asynchronous mode is required for reordering messages</b>! Specifically, messages in Trap may be
427      * reordered by the transports due to race- or network conditions. Reordering these messages – and redispatching
428      * them – may cause dropped or disconnected network connections, so Trap will NOT reorder incoming messages if in
429      * synchronous mode.
430      * 
431      * @param async
432      *            The new mode. True makes the endpoint asynchronous, while false disables it.
433      */

503      * individual configuration the transports may have had. The inverse does not apply.
504      * <p>

517      *            The new maximum number of transports.
518      *            <p>
519      *            Note that 0 (zero) is a permitted value for maximum number of transports. This will not close the Trap
520      *            connection, but cause a non-deterministic sequence of transport opening and closings, causing an
521      *            abnormal load on client, server and network, but making it difficult to predict which transport (and
522      *            thus connection) will be used at any given point in time. This can be used to mitigate suspected MITM
523      *            attacks, but is not recommended otherwise. This connection sequence is not guaranteed secure. <i>This
524      *            (zero) functionality is for experimental purposes, and may be removed at any time!</i>

578     /**
579      * Fetches all the available authentication context, for all transports; including, where available, IP numbers,
580      * ports, etc. This context will be based on {@link TrapContextKeys}, among other detail information, and consist of
581      * a map for each transport, indexed by transport name. This information will provide detail level information on
582      * the connection for each transport.
583      * <p>
584      * Typical output for a single-transport scenario may look as follows:
585      * 
586      * <pre>
587      * {
588      *  socket= {
589      *      LastAlive=1407198698446,
590      *      Format=REGULAR, 
591      *      State=AVAILABLE, 
592      *      Transport=socket/1/AVAILABLE/704f459c, 
593      *      LocalIP=127.0.0.1, 
594      *      LocalPort=59277, 
595      *      Configuration=trap.enablecompression = true
596      *      RemotePort=59278, 
597      *      Protocol=tcp, 
598      *      RemoteIP=127.0.0.1, 
599      *      Priority=-100
600      *      }
601      *  }
602      * </pre>
603      * 
604      * From this information, we can gather all pertinent information about the transport's internal state. This is a
605      * socket transport that is not configured to connect, so was provided by a server endpoint. The state is AVAILABLE,
606      * and lastAlive specifies when there was traffic seen. We have the TCP 5-tuple information, which can be used to
607      * identify a connection that has not changed.
608      * <p>
609      * This information is generally available to a {@link TrapAuthentication} instance assigned to the endpoint and/or
610      * transports. The authentication will be automatically invoked whenever any changes may occur on the underlying
611      * connection, such as a new TCP session, unsecure packet, new HTTP request, transport switchover, or a new
612      * transport being introduced. <b>If this information is needed to provide <i>security</i>, TrapAuthentication
613      * should be used</b>.
614      * <p>
615      * This method is intended for <i>informational</i> purposes. The returned values are immediately polled, will
616      * include all possible values, and will be requested synchronously; the CPU cost of using this method often is
617      * non-negligible. The method is thread safe with no need for synchronization.
618      * 
619      * @return A map of all authentication context available.
620      * @since 1.1.5
621      */
622     public Map<String, Map<String, Object>> getTransportAuthenticationContexts();
