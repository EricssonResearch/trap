38  import com.ericsson.research.trap.TrapEndpoint;
39  import com.ericsson.research.trap.TrapException;
40  import com.ericsson.research.trap.spi.TrapMessage;
41  
42  /**
43   * A FIFO queue for messages. Implementations of MessageQueue must provide a thread-safe, FIFO queue of messages. Some
44   * implementations may block when asked to put a message, others may block.
45   * <p>
46   * The specific properties of the queue affect the performance, throughput and memory footprint of Trap. For example, a
47   * <i>blocking</i> queue would wait for messages to be sent before sending new ones. It is useful in a high-throughput
48   * environment to prevent the buffer from growing endlessly due to a slight mismatch in processing speed. An
49   * array-backed buffer is good for static memory usage but may require too much memory for certain applications.
50   * <p>
51   * Trap ships with a number of built-in buffer types, described below. All of these classes are defined in the package
52   * <i>com.ericsson.research.trap.impl.queues</i> in the <i>trap-core</i> maven artefact.
53   * <ul>
54   * <li><b>ArrayBlockingMessageQueue</b> is an array-backed, blocking queue, with a limit based on the number of messages
55   * in the queue. Useful for static memory usage in a high-throughput environment. The size of an Array-backed queue
56   * needs to be chosen with consideration; too low and not enough messages will fit and the queue will block too often.
57   * Too high, and the queue uses too much RAM from the JVM.
58   * <li><b>LinkedBlockingMessageQueue</b> is a LinkedList-backed, blocking queue, with a limit of number of messages. A
59   * good, low-memory, blocking queue type where Trap does not need to monitor byte allocations. Selected by
60   * {@link TrapEndpoint#BLOCKING_MESSAGE_QUEUE} if no better linked queue is present.
61   * <li><b>LinkedByteMessageQueue</b> is a LinkedList-backed queue that computes its size based on the number of bytes in
62   * the queue, as opposed to the number of messages. This is a generally slower queue type from a computational
63   * perspective, but can be useful to prevent an application from oversaturating the network on slower networks. The
64   * application can then verify how much data is queued for sending. Default implementation for
65   * {@link TrapEndpoint#REGULAR_BYTE_QUEUE}.
66   * <li><b>LinkedByteBlockingMessageQueue</b> is the same as a LinkedByteMessageQueue except it blocks on the number of
67   * bytes as well, providing a throttling mechanism. This provides a way to ceiling the amount of data Trap can and will
68   * accept. Default implementation for {@link TrapEndpoint#BLOCKING_BYTE_QUEUE}.
69   * <li><b>LinkedMessageQueue</b>. The simplest of queue types, this provides a LinkedList-backed queue with no
70   * limitations. This is the lowest-overhead queue, but provides no throttling mechanisms. Default implementation for
71   * {@link TrapEndpoint#REGULAR_MESSAGE_QUEUE}.
72   * </ul>
73   * <p>
74   * The following queues are available in the <i>trap-ext-queues-15</i> maven artefact:
75   * <ul>
76   * <li><b>CLQMessageQueue</b> is a ConcurrentLinkedQueue-based blocking queue, requiring Java 1.5 or later. It provides
77   * high-throughput, variable-memory queue based on java.util.concurrent, but has a high number of object
78   * allocation/deallocations. Replaces LinkedBlockingMessageQueue as the default implementation of
79   * {@link TrapEndpoint#BLOCKING_MESSAGE_QUEUE}.
80   * <li><b>CABMessageQueue</b> is an ArrayBlockingQueue-backed queue. It provides the highest sustained throughput of the
81   * default Trap message queues, at the expense of requiring a fixed amount of memory.
82   * </ul>
83   * <p>
84   * The default queue chosen by Trap will be either CLQMessageQueue or LinkedBlockingMessageQueue, with a fairly generous
85   * limit of 1000 messages. This should provide high throughput on most situations, with latency determined by network.
86   * If low latency (or not oversaturating the network) is required, Trap should be reconfigured to use a different
87   * message buffer. A custom buffer may also be included, as long as it conforms to this interface, especially the
88   * thread-safe part.
89   * 
90   * @author Vladimir Katardjiev
91   */
92  public interface MessageQueue
93  {
94      
95      /**
96       * Inserts a TrapMessage at the end of the queue
97       * 
98       * @param m
99       *            The message to insert
100      * @throws TrapException
101      *             If the message could not be inserted
102      */
103     public void put(TrapMessage m) throws TrapException;
104     
105     /**
106      * Looks at the first message in the queue
107      * 
108      * @return The first message of the queue
109      */
110     public TrapMessage peek();
111     
112     /**
113      * Removes the first message of the queue
114      * 
115      * @return The removed message
116      */
117     public TrapMessage pop();
118     
119     /**
120      * Current length of the queue.
121      * 
122      * @return The current length of the queue
123      */
124     public int length();
125     
126     /**
127      * Maximum size of the queue
128      * 
129      * @return The maximum size of the queue.
130      */
131     public long size();
132     
133     /**
134      * Accessor method for the general message queue type provided by this queue, as defined in {@link TrapEndpoint}.
135      * This is no longer used by Trap, instead preferring to use {@link #createNewQueue()}. The method is retained for
136      * debugging purposes.
137      * 
138      * @return A string representing the queue type name.
139      */
140     public String getQueueType();
141     
142     /**
143      * Method to ask if the queue has more than one elements in it
144      * 
145      * @return <i>true</i> if the queue has more than one element, <i>false</i> otherwise.
146      */
147     public boolean hasMoreThanOne();
148     
149     /**
150      * Creates a new queue with the same settings as this queue, but none of the messages.
151      * 
152      * @return The new queue. It should be of the same class as the old queue, with the same settings.
153      * @since 1.1
154      */
155     public MessageQueue createNewQueue();
156     
157 }
